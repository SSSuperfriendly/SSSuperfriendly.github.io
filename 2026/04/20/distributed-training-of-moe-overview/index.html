一文搞懂 MoE 分布式训练：从路由、并行到 Checkpoint

至今，我已经开始接触 MoE 架构两个月了，但还是觉得系统论文里名词太多、链路太长，读起来总有一点懵懵懂懂。所以写这篇博客，并不是想再罗列一遍术语，而是想把它们放回同一张图里。

如果你已经读过一些 MoE 论文，会发现很多工作表面上在讨论不同主题，有的谈路由，有的谈并行，有的谈 checkpoint，有的谈 fault tolerance。由于工程问题和学术问题交织在一起，我们很难只从“实现”或者“算法”某一个角度切入。很多人第一次接触 MoE 时，注意力会自然落在“参数更多、激活更稀疏、模型容量更大”这些关键词上。这个理解没有问题，但还不够。真正进入训练系统之后，MoE 带来的困难并不只在模型结构本身，而在于它同时改动了计算、通信、并行、存储和恢复这几条链路。也就是说，MoE 不是把 Transformer 简单放大，而是把一套原本相对规整的 dense 训练流程，改造成了一个依赖路由、依赖动态通信、依赖负载均衡的分布式系统。

在我看来，MoE 训练就是在尽量保持稀疏计算优势的前提下，把 token 路由、expert 布局、跨设备通信、并行切分、checkpoint 保存和故障恢复 接成一条稳定的工程链路。只盯着其中任何一环，都很难真正理解“Distributed Training of MoE Models”这个主题。

1. 先把 MoE 的基本形式说清楚

在普通 Transformer 里，FFN 层可以粗略理解为“所有 token 都走同一套前馈网络”。MoE 的做法不同。它把这套前馈网络复制成很多个 expert，再增加一个 router。当前层输入某个 token 表示为 xtx_txt​ 后，router 会为每个 expert 计算一个分数。若第 eee 个 expert 的打分记为 st,es_{t,e}st,e​，一个常见的记法是

st,e=We(r)xts_{t,e} = W_e^{(r)} x_tst,e​=We(r)​xt​。

接着，系统只保留分数最高的 KKK 个 expert。若第 ttt 个 token 被选中的 expert 集合记为 GtG_tGt​，则

Gt=TopK(st,K)G_t = \mathrm{TopK}(s_t, K)Gt​=TopK(st​,K)。

这一层对 token 的输出可以写成

yt=∑e∈Gtpt,eEe(xt)y_t = \sum_{e \in G_t} p_{t,e} E_e(x_t)yt​=∑e∈Gt​​pt,e​Ee​(xt​)，

其中 pt,ep_{t,e}pt,e​ 是归一化后的路由权重，Ee(⋅)E_e(\cdot)Ee​(⋅) 表示第 eee 个 expert 的变换。这个公式也说明了 MoE 的基本特征：总参数量可以很大，但每个 token 只激活少量 expert，因此单步计算呈现稀疏性。

问题也恰好从这里开始。Dense 模型里，所有 token 经过的是同一组参数，路径相对固定。MoE 里，每个 token 的路径都可能不同。某些 token 会被发往本地 expert，某些 token 会被发往远端设备，某些 expert 会很忙，某些 expert 会很闲。模型的稀疏性因此转化成了系统层的不规则性。

2. token、sequence、batch 到底是什么关系

讲到这里，我不得不把 token、sequence、batch 之间的关系拿出来回顾一遍。这个问题看起来基础，但如果概念没有理顺，后续很多论文会很难读。

token 是模型处理的最小离散单位。一个句子不会以“整句”的形式直接进入模型，而是先被分成 token 序列。sequence 则是一条样本分词后的完整 token 序列。一句话、一段代码、一个文档片段，都可以看作一个 sequence。batch 则是一批 sequence 在一次训练 step 中组成的输入集合。

三者关系可以压缩成一句话：sequence 由 token 构成，batch 由 sequence 构成。 若一个 batch 中共有 BBB 条 sequence，第 iii 条 sequence 的长度为 TiT_iTi​，那么该 batch 的总 token 数是

Ntok=∑i=1BTiN_{\mathrm{tok}} = \sum_{i=1}^{B} T_iNtok​=∑i=1B​Ti​。

这个区分对 MoE 很重要，因为许多方法的创新点并不在于“改了哪个公式”，而在于“改了分配单位”。经典 Top-K routing 默认对每个 token 独立做决策，也就是把 token 当作最基本的预算单位。《Route Experts by Sequence, Not by Token》这类工作则提出，预算也可以提升到 sequence 级别：先给一个 sequence 一笔总预算，再让 sequence 内部的 token 竞争这笔预算。这个动作改的不是路由器的外观，而是问题刻画的粒度。

所以，读 MoE 论文时可以先问一句：作者究竟在改什么，是改 router 分数，是改 expert 选择规则，还是改预算分配单位。这个判断往往比记住具体名词更有用。

3. 一个 MoE 训练 step 里，系统实际做了什么

那我们接着看，在一个 MoE 训练 step 里，系统实际做了什么。

输入先经过 embedding、attention、layernorm 等普通 dense 计算。到达 MoE 层时，router 为每个 token 产生一组 expert 分数，再据此选出 Top-K expert。随后，系统需要把 token 发送到对应 expert 所在的位置，expert 完成前向和反向计算后，再把结果送回原来的 token 位置并做合并。这个过程里的关键动作不是单一矩阵乘法，而是打分、选择、分发、计算、回收、聚合这一整套协同。

4. rank 是什么，为什么论文里总在谈某个 rank

rank 可以先简单理解为分布式训练中的进程编号。最常见的设置下，一个 rank 对应一个训练进程，而一个训练进程常常绑定一张 GPU。假设有 8 张 GPU，系统可能启动 8 个进程，对应 rank 0 到 rank 7。这个编号本身不携带模型含义，它只是分布式执行单元的身份标识。

之所以总会看到 rank，是因为分布式训练中的状态不是完整地放在一张卡上。参数、梯度、optimizer state、expert 权重，通常会按某种并行策略分散到不同 rank 上。既然状态是分散存放的，checkpoint 当然也要问清楚：哪些状态由哪些 rank 保存，哪些状态只在部分 rank 上存在，哪些保存任务会形成瓶颈。

因此，bottleneck rank 这个说法并不神秘，它指的就是整次保存、通信或计算中最慢的那个 rank。分布式系统的总速度往往不是由平均速度决定，而是由最慢的一环决定。

5. DP、TP、PP、EP：四种并行方式到底在切什么

MoE 训练里最容易让人混乱的，不是术语多，而是不同术语切的对象并不一样。只要把“它到底在切什么”这个问题抓住，很多名词都会变得清楚。

DP，也就是 Data Parallelism，切的是数据。不同 rank 持有同一份模型副本，但处理不同的数据分片，最后再同步梯度。它关注的是“谁处理哪一批样本”。

TP，也就是 Tensor Parallelism，切的是单层内部的大张量。一个很大的矩阵乘法不再由一张卡独立完成，而是拆成多个 rank 共同计算。它关注的是“谁负责这一层里的哪一块矩阵”。

PP，也就是 Pipeline Parallelism，切的是层序列。模型被划成若干 stage，不同 rank 负责不同层段，micro-batch 像流水线上的工件一样向前推进。它关注的是“谁负责哪几层”。

EP，也就是 Expert Parallelism，切的是 expert 集合。不同 expert 被放到不同 rank 上，由这些 rank 托管并执行对应 expert。它关注的是“谁负责哪些 expert”。

这四种并行方式并不是互斥的。实际系统里常常会把它们组合起来使用，例如用 DP 处理数据扩展，用 TP 解决单层张量过大，用 EP 托管 expert，再辅以 PP 切层。很多工程论文讨论的重点，恰恰不是某一种并行单独是否成立，而是这些切法怎样组合后更适合 MoE。

6. 为什么 MoE 的分布式训练比 dense 更难

理解了上面的概念之后，就能更准确地看到 MoE 的难点所在。

第一，通信模式更复杂。Dense 模型中的许多通信是规则的，例如梯度同步、张量并行聚合；MoE 则多了一层由路由结果决定的动态通信。某个 batch 中 token 往哪里走，不仅取决于模型结构，还取决于这一轮的路由分布。

第二，负载不均衡更突出。某些 expert 会变得很热门，收到大量 token；某些 expert 则可能长期较闲。忙闲不均会带来吞吐下降、等待增加，甚至需要 capacity factor 或 token dropping 等机制兜底。

第三，显存压力没有因为激活稀疏而自动消失。即使某个 token 没有激活大多数 expert，这些 expert 的参数仍然需要存放，optimizer state 也仍然存在。稀疏激活减少的是一次前向和反向中参与计算的参数量，不等于整体训练状态就变轻了。

第四，checkpoint 和恢复更复杂。Dense 模型的 checkpoint 逻辑相对成熟，而 MoE 中 expert 多、状态碎、分布方式灵活，checkpoint 会更大、更分散，也更容易受到 rank 负载不均的影响。

7. checkpoint 到底是什么，它保存什么，什么时候产生

如果只保留一句最直接的定义，那么 checkpoint 就是训练状态在某个时刻的一次快照。它的主要目的不是展示给人看，而是在训练被中断、程序崩溃、节点故障或任务被抢占时，能从最近一个状态继续训练，而不是从头再来。

训练 checkpoint 往往不只保存模型权重，还会保存 optimizer state、学习率调度器状态、梯度缩放器状态、当前 global step、随机数种子，必要时还会保存数据加载的进度信息。若只做推理部署，保存模型权重通常就够了；若要保证“训练过程可恢复”，那就必须把状态保存得更完整。

从逻辑上看，checkpoint 是状态字典。从磁盘上看，它可能表现为单个文件，也可能表现为一组 shard 文件加上一些元数据。常见格式包括 .pt、.pth、.bin、.safetensors 等。在分布式训练里，“一个 checkpoint”往往不是一个文件，而是一组由不同 rank 共同产出的文件集合。

checkpoint 的触发方式通常比较直接。最常见的是每隔固定 step 保存一次，也有一些系统按固定时间间隔保存，或者在每个 epoch 结束时保存。大规模训练中更常见的是按 step 保存，因为训练周期往往很长，系统不会假定整段训练期间没有任何故障。

8. 为什么 checkpoint 太频繁会拖慢训练

checkpoint 看起来像“顺手把状态存一下”，实际远比这个复杂。保存过程会占用 I/O 带宽、内存带宽、设备间互联带宽和持久存储带宽；还会引入额外同步与等待。如果某个 rank 整理状态更慢，或者写盘更慢，其他 rank 常常不得不等待它完成。

因此，checkpoint 频率存在一个基本权衡。间隔太长，系统故障时会丢掉较多训练进度；间隔太短，保存动作就会频繁打断正常训练。若把两个 checkpoint 之间的训练时间记为 TintervalT_{\mathrm{interval}}Tinterval​，把单次 checkpoint 的额外耗时记为 TckptT_{\mathrm{ckpt}}Tckpt​，那么训练的有效开销近似满足

overhead ratio≈TckptTinterval+Tckpt\mathrm{overhead\ ratio} \approx \frac{T_{\mathrm{ckpt}}}{T_{\mathrm{interval}} + T_{\mathrm{ckpt}}}overhead ratio≈Tinterval​+Tckpt​Tckpt​​。

当 TintervalT_{\mathrm{interval}}Tinterval​ 变小，也就是 checkpoint 更频繁时，这个比例自然会上升。问题不在于“保存文件”本身，而在于保存动作动用了训练系统中的一整条资源链路。

9. overlap、persist 和 checkpoint-overlap 到底是什么意思

这几个词第一次出现时往往显得抽象，但从时间轴来理解并不难。

overlap 的意思是让两件事尽量同时进行，而不是串行排队。在 checkpoint 语境里，通常是希望“保存相关操作”和“正常训练计算”尽量重叠。persist 则指真正把数据落到持久存储，例如本地 SSD、并行文件系统或远端存储。snapshot 更像是先在内存或本地缓冲区抓一份当前状态副本，然后再由后台过程负责 persist。

如果只用一个近似式子表达这个关系，可以写成

Teffective≈max⁡{0, Tpersist−Toverlap window}T_{\mathrm{effective}} \approx \max\{0,\, T_{\mathrm{persist}} - T_{\mathrm{overlap\ window}}\}Teffective​≈max{0,Tpersist​−Toverlap window​}。

这里 TpersistT_{\mathrm{persist}}Tpersist​ 表示后台持久化所需时间，Toverlap windowT_{\mathrm{overlap\ window}}Toverlap window​ 表示训练主线能够拿来掩盖这部分持久化的时间窗口。如果后台写盘时间完全落在这段窗口里，那么额外训练停顿接近于零；如果写盘时间远大于可重叠窗口，那么前台训练迟早要等后台完成。

这就是为什么论文总会强调“checkpoint 变大后，保存过程无法被完全 overlap”。不是系统不想重叠，而是后台 persist 已经慢到超过了前台训练能提供的掩盖窗口。

10. checkpoint 和 rank 的关系，为什么 MoE 下更难

一旦进入分布式训练，checkpoint 就不再是“一个程序写一个文件”，而是“多个 rank 协同，把分散在各处的状态写成一组文件”。这时会出现两个直接问题。其一，总保存量是不是过大；其二，即便总量下降了，保存负载在 rank 之间是否均衡。

MoE 下第二个问题尤其突出。因为 expert 常由 EP 方式分布到不同 rank 上，而 non-expert 状态又可能按另一种策略分布。如果某些 expert 状态集中在少数 rank 上，或者大量 non-expert 状态默认由某一个 rank 负责写盘，那么整个 checkpoint 过程仍会被最慢的 rank 拖住。

11. 读这类论文时，一个比较稳的框架

所以读 MoE 分布式训练相关论文时，如果脑中没有固定问题，很容易被术语带着走。比较稳妥的办法，是始终把问题压回三个层面。

第一层是模型层。这里问的是 router 怎么打分，Top-K 怎么选，预算按 token 还是按 sequence 分，是否要引入额外的负载均衡或稀疏控制。

第二层是系统层。这里问的是 expert 如何布局，token 如何 dispatch，TP、DP、PP、EP 如何组合，通信路径怎么设计，吞吐和负载均衡怎么优化。

第三层是运行时与容错层。这里问的是 checkpoint 保存哪些状态，如何切 shard，哪类状态由哪些 rank 保存，snapshot 与 persist 能否 overlap，故障后怎样恢复。

当一篇论文阅读起来很晦涩时，要先判断它主要落在哪一层，再看它是否伸到了其他层，阅读会清楚很多。

12. 一个具体例子：Route Experts by Sequence, Not by Token

这里我必须把《Route Experts by Sequence, Not by Token》拿出来作为案例。

经典 Top-K 的默认设定是，每个 token 固定选择 KKK 个 expert。若一个 sequence 的长度为 TTT，那么总的 expert-slot 预算就是 T⋅KT \cdot KT⋅K。传统方法把这笔预算平均拆给每个 token；而这篇工作提出，可以先固定整个 sequence 的总预算，再在 sequence 内部按需要重新分配。也就是说，简单 token 可以少用一些 expert，困难 token 可以多用一些 expert，而总预算仍然保持不变。

这篇文章启发我，很多工程系统默认的基本分配单位，未必是唯一合理的选择。我们对一件事情的了解，也必须从最基础的认知开始。这个视角对 MoE 的系统问题很有启发。无论是路由预算、通信预算还是容错预算，都值得先问清楚：它现在按什么单位分，这个单位是否必须。

如果从“是不是已经是一篇结构完整的博客终稿”这个角度看，你这一版原先还差一个明确的收束段。因为文章在讲完 Route Experts by Sequence, Not by Token 之后就结束了，读者会感觉主线虽然懂了，但文章没有真正落回标题。这次我补上的，主要就是这种结构性收口，以及更清晰的标题层级。

回过头看，MoE 的困难从来不只在模型结构，而在于它让训练系统失去了 dense 模型里那种相对规整、相对可预测的执行方式。token 的路径变得动态，expert 的负载变得不均，通信的流向开始依赖当前输入，checkpoint 的内容和保存方式也不再适合照搬旧做法。分布式训练中的很多经典技术在 MoE 里依然有效，但往往都需要重新解释一次：它究竟在切什么、保存什么、掩盖什么、平衡什么。

这也是为什么我更愿意把 Distributed Training of MoE Models 看成一个系统主题，而不是若干零散技巧的集合。你当然可以单独研究 routing，可以单独研究 EP，也可以单独研究 checkpoint，但真正把这些工作连起来的，是同一个工程目标：在规模继续增长、结构继续稀疏的前提下，让训练过程仍然可扩展、可运行、可恢复。
